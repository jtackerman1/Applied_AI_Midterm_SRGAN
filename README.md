# Applied AI Midterm: Transfer Learning with SRGAN Images

## Project objective

This project will build a reproducible PyTorch experiment comparing two binary image classifiers:

- **Model A:** a transfer-learning classifier trained on original images resized to 128×128.
- **Model B:** the same classifier architecture trained on 128×128 images generated by an SRGAN from 32×32 inputs.

This first phase contains only the project foundation: repository structure, dependencies, shared configuration, documentation, and configuration tests. Data processing, model definitions, training, evaluation, and results will be added in later phases.

## Planned experimental flow

1. Validate a two-class dataset and discover its image files.
2. Create one stratified 70% training / 30% testing split with random seed 42.
3. Persist paths and labels so every notebook uses the same reserved examples.
4. Train Model A on original training images resized to 128×128.
5. Train the SRGAN only on training examples, mapping 32×32 inputs to 128×128 targets.
6. Generate 128×128 training images for Model B with the trained generator.
7. Train Model B on the generated training images.
8. Evaluate both classifiers using corresponding examples from the same reserved test split.
9. Compare accuracy, precision, recall, F1, ROC AUC, classification reports, confusion matrices, and ROC curves.

Test images must never train either classifier or the SRGAN. This repository does not claim experiment results before those experiments have actually run.

## Planned notebooks

The `notebooks/` directory is intentionally empty in this foundation phase. Later phases will add lightweight notebooks that call reusable modules under `src/applied_ai_midterm/`:

1. `01_data_preparation.ipynb` — inspect data and persist the stratified split.
2. `02_train_classifier_original.ipynb` — train Model A.
3. `03_train_srgan.ipynb` — train and checkpoint the SRGAN.
4. `04_generate_srgan_images.ipynb` — create Model B inputs.
5. `05_train_classifier_generated.ipynb` — train Model B.
6. `06_evaluate_and_compare.ipynb` — evaluate both models and generate reports.

## Dataset placement

Place the original images under `data/raw/` with exactly one directory per class:

```text
data/raw/
├── class_a/
│   ├── image_001.jpg
│   └── image_002.jpg
└── class_b/
    ├── image_003.jpg
    └── image_004.jpg
```

Do not commit dataset files. Future processed data belongs in `data/processed/`, and future generator outputs belong in `data/generated/`. Git ignores the contents of all three locations while retaining their `.gitkeep` placeholders.

## Google Colab workflow

Open [notebooks/00_colab_setup.ipynb](notebooks/00_colab_setup.ipynb) in Colab and select
**Runtime → Change runtime type → T4 GPU** before running it. The setup notebook:

1. Mounts Google Drive.
2. Clones or updates this GitHub repository.
3. Keeps Colab's GPU-enabled PyTorch installation.
4. Links data, splits, generated images, checkpoints, and artifacts to persistent Drive storage.
5. Verifies GPU and dataset availability.
6. Creates or reloads the seed-42 stratified split.
7. Runs the test suite.

Upload the dataset to this Drive layout before the dataset-verification cell:

```text
MyDrive/Applied_AI_Midterm_SRGAN_runtime/
└── data/
    └── raw/
        └── train/
            ├── cats/
            └── dogs/
```

The downloaded dataset's `validation/` and `test/` siblings may also be uploaded, but the project
ignores them and creates its own single 70/30 split from `raw/train`. Do not upload the local
`data/splits/*.csv` files because they contain machine-specific absolute paths; Colab recreates the
same assignments with Drive paths.

Run notebooks `01` through `06` in order. Notebook 03 is the long-running stage. Its five-epoch
checkpoints are stored in Drive, so after a Colab disconnect, rerun notebook 00 and then notebook 03
to resume. Generated images and final result tables are also stored under the persistent Drive root.

Never place Drive, Kaggle, or Colab credentials inside the repository.

PyTorch and torchvision are the planned deep-learning libraries. TensorFlow is not a dependency.

## Configuration

Shared defaults live in `configs/config.yaml` and can be loaded with:

```python
from applied_ai_midterm.config import load_config

config = load_config()
print(config.random_seed)
```

The loader uses `pathlib`, validates required and unknown keys, checks the split ratios and 4× resolution scale, and provides a clear error when the configuration file is missing.
